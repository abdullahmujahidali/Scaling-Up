System Design · Lesson 02

Consistency: how fresh is the data you read?

One idea, taught slowly, with a story. (Failover and the "nines" are now their own separate lessons.)

~8 min One concept: consistency Win: explain weak vs eventual vs strong in plain words
If you remember one thing Consistency = "after someone writes data, when does everyone else see it?" Sometimes instantly, sometimes after a short delay, sometimes never. That's the whole idea.

Start with a story (no tech yet)

Imagine a big office with one whiteboard in the main hall and a few copies of it in side rooms. Someone walks up to the main board and writes:

"Lunch moved to 1pm."

Now — the people in the side rooms are reading their copies, not the main board. So here's the only question that matters:

When do the copies in the side rooms get updated?

There are three honest answers, and each one is a real way real systems behave. That's it — that's consistency. Let's walk the three.

1 · Weak — "the copies might never catch up"

The story: You write "Lunch at 1pm" on the main board. Nobody ever bothers to update the side-room copies. If you happened to not see the change, it's just gone for you. No promises.

Why would anyone want this? Because it's the fastest, and sometimes old info simply doesn't matter. Think of a live video call: if half a second of audio drops, you don't rewind and replay it — that moment is gone and that's fine. You care about now, not catching up on the past.

Quick check: in plain words, what does weak consistency promise about a read after a write?
Basically nothing — the read might not see the write, ever. It's best-effort. We accept that in exchange for speed, when stale/missed data is harmless (like a live call).

2 · Eventual — "the copies catch up in a moment"

The story: You write on the main board. An assistant walks around and updates each side-room copy — it takes a few seconds, but eventually every copy says "1pm." If someone reads a side-room board in that gap, they briefly see the old time. Then it fixes itself.

Everyday version: you change your home address with the post office. It's not updated everywhere the instant you submit it — it ripples out over a bit of time. Eventually all the systems agree.

Django bridge — this is the one to really get You've done this without naming it. A single Postgres is the main board. The moment you add a read replica (a copy database to take read load off the main one), you've added a side-room board. A user updates their profile (writes to the main board), instantly reloads, and the page reads from the replica — which hasn't caught up yet — so they see the old name for a second. That gap is called replication lag, and it's exactly eventual consistency. You didn't break anything; you chose this the moment you added a replica.
Quick check: under eventual consistency, will the read eventually see the write? And what's the catch?
Yes — it'll see it, just not instantly. The catch: for a short window (often milliseconds), a read can return the old value before the copies catch up.

3 · Strong — "every board changes at the exact same instant"

The story: The rule is now strict — the change isn't considered "done" until every board, main and side rooms, has been updated together. So the moment anyone can read "1pm" anywhere, it says "1pm" everywhere. No one ever sees the old time.

Why pay for this? Because for some data, being wrong for even a second is unacceptable. Your bank balance is the classic example: if you withdraw $100, every part of the system must agree your balance dropped — instantly. You'd never want a "side room" still showing the old, higher balance.

The two words that decide it Synchronous = "wait until all copies agree before we call the write done" → that's how you get strong. Asynchronous = "save it now, copy to the others later" → that's how you get eventual. Same two words show up all over system design — you just learned them with a whiteboard.

Let's pull it together

Cover the answer. Name the three levels weakest → strongest, in one phrase each, using the whiteboard story.
Weak — side rooms might never catch up (read may never see the write). Live video call.
Eventual — side rooms catch up in a moment (read sees it after a short delay). Read replica / changing your address.
Strong — every board changes at once (read always sees it instantly). Bank balance.

One quick quiz

Just one — using the exact Django situation from the bridge above.

You added a read replica to take load off your Postgres. A user updates their profile name, instantly reloads, and sees the old name for one second, then it corrects. Which consistency is this?
Your win today You can now explain consistency with a story: it's just "when do the side-room copies catch up?" — never (weak), soon (eventual), or instantly (strong). And you know the real-world version: add a read replica to Django and you've chosen eventual. That's one solid idea, fully yours. That's plenty for one sitting.
